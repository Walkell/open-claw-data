# towney 投委会架构说明

> 整理时间：2026-06-08
> 邀请目的：寻求外援辅助，优化投委会运行效率

---

## 一、背景

towney 是一个基于 AI Agent 的投资分析决策框架。目前已建立"投委会"架构，包含多个角色的 Agent 协作，但在实际运行中存在以下问题：

1. **CIO 集权**：首席投资官一人做了所有分析决策，子 Agent（分析师）形同虚设
2. **流程不闭环**：规定的"派发子Agent → 汇总 → 决策"流程从未完整执行过
3. **职责模糊**：哪些场景调子Agent、哪些不调，没有清晰规则

希望外部视角帮助诊断架构问题，给出可落地的优化建议。

---

## 二、当前角色定义

### 成员清单

| 角色 | 实体 | 职责 | 决策权 | 工具 |
|------|------|------|--------|------|
| **CIO** | 主 Agent | 汇总子Agent报告、做最终投资判断 | ✅ 唯一切实建议权 | sessions_spawn |
| **Table Desk** | 主 Agent（同一实体） | Bitable读写、行情拉取、交易记录、盯盘 | ❌ 无 | feishu_bitable_*, akshare_*, exec |
| **Research** | 子 Agent | 技术/基本面/估值/财报 四维评分 | ❌ 只分析 | 暂无独立工具（靠 CIO 喂数据） |
| **Industry** | 子 Agent | 宏观（利率/PMI/社融）+ 行业景气度 | ❌ 只分析 | 暂无独立工具 |
| **News** | 子 Agent | 事件提取+情绪+置信度 | ❌ 只分析 | web_search/tavily_search |
| **Risk** | 子 Agent | 综合风险评分+否决 | ✅ VETO（≥7/10） | 暂无独立工具 |

### 关键约束

- **CIO 和 Table Desk 是同一个 Agent 的两种模式**：平时是 Table Desk（只读写不决策），需要决策时切为 CIO（召投委会）
- **Risk 拥有独立否决权**：组合风险评分 ≥7/10 → 自动 VETO，禁止新增买入。CIO 可 override，但必须记录理由
- **News 禁止输出交易建议**：只提取事件+情绪+置信度，不能出 BUY/SELL/HOLD
- **Table Desk 是唯一写 Bitable 者**：子 Agent 不能直接写 Bitable

## 三、目标决策流程（设计中的，未真正执行过）

```
用户指令
  ↓
┌─ 简单操作（行情/表格读）→ Table Desk 自己处理
│
└─ 涉及投资决策 → CIO 召投委会：
    ├── Research ─── 四维评分
    ├── Industry ─── 宏/行双维
    ├── News ─────── 事件+情绪
    └── Risk ─────── 风险评分（否决权）
          ↓
    CIO 汇总四票 → 出最终建议
          ↓
    Table Desk 写入 Bitable / 交易记录
```

### 强制调用规则

| 场景 | 必须调用 | 豁免 |
|------|----------|------|
| 新增观察/建仓判断 | Research + Industry + Risk | — |
| 持仓减仓/止损 | Research + Risk | — |
| 大跌应对 | Research + Risk | 开盘5分钟加仓扫描 |
| 定期复盘 | 四个全调 | — |
| 简单查询 | 不调 | 全豁免 |
| 执行明确指令 | 不调 | 全豁免 |

## 四、实际运行情况

### 真实发生过的事

| 事件 | 日期 | 流程 | 结果 |
|------|------|------|------|
| 全持仓复盘 | 6/2 | 完整走了一遍：CIO 派发 4 子Agent → 汇总决策 | ✅ 唯一一次正常流程 |
| 澜起科技减仓 | 6/2 | 投委会决议减仓 20pp @237 | ❌ 6/3 暴涨 +7.13%，踩在起涨前 |
| 大跌开盘没加仓 | 6/8 | CIO 优先做宏观归因，没及时扫描加仓机会 | ❌ 错过海光/澜起/华海的低吸窗口 |

### CIO 实际做了但不该做的

- 拉行情数据（应属 Table Desk 或子 Agent 自取）
- 搜新闻、做行业分析（应属 News/Industry）
- 自己打分估值（应属 Research）
- 综合判断风险（应属 Risk）
- 写出了全持仓的分析结论（没有子 Agent 输入）

### 子 Agent 实际没做的

- 除了 6/2 那次，其他时间从没被调用过
- 没有独立工具，数据全靠 CIO feed
- 没有定期驻守在跑

## 五、平台能力

### 当前可用工具

| 类别 | 工具 | 说明 |
|------|------|------|
| Bitable 读写 | `feishu_bitable_*` (7个) | 飞书多维表格，持仓/交易/观察/报告表 |
| A股行情 | `akshare__get_realtime_data` | 实时行情（eastmoney/腾讯财经） |
| A股历史 | `akshare__get_hist_data` | K线/技术指标 |
| A股财报 | `akshare__get_financial_metrics/income/balance/cash` | 三张表+关键指标 |
| A股新闻 | `akshare__get_news_data` | 个股新闻 |
| 搜索 | `web_search`, `tavily_search`, `web_fetch` | 通用+财经搜索 |
| 子Agent | `sessions_spawn`, `sessions_yield` | 派发独立子会话 |
| Shell | `exec` | 执行系统命令、curl 等 |

### Agent 平台特性

- 主 Agent 和子 Agent 共享工作区文件系统
- 子 Agent 通过 `sessions_spawn` 创建，可指定 `tools_allow` 限制工具权限
- 支持 `sessions_yield` 等待子 Agent 完成后汇总
- 支持 cron 定时任务

## 六、核心痛点

1. **CIO 集权**：一人身兼 CIO + Table Desk + 分析师，没有制衡
2. **子 Agent 闲置**：工具不分配、不驻守，召一次嫌麻烦
3. **职责边界不物理**：CIO 和 Table Desk 是同一实体，切换靠自觉
4. **流程没演练过**：从"用户一句话"到"投委会出结论"的完整链路从未真正跑通
5. **工具分配不对**：数据应该子 Agent 自取，现在是 CIO 取了喂

---

## 七、求助方向

1. **架构诊断**：当前角色分工设计是否合理？是否应该把 CIO 和 Table Desk 拆成两个独立 Agent？
2. **工具分配**：子 Agent 各自该有哪些工具？如何避免并发写入冲突？
3. **流程固化**：如何让"强制调用子Agent"成为物理约束而非自觉？
4. **落地路径**：如何用最小改动让这条链真正跑通一次？
