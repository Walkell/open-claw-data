# custom-config-maintain · 配置档模板同步维护

> **背景**：`CONFIG_SAMPLE.md` 是模板，不被任何 agent 读取；`CONFIG_{PRINCIPAL_ID}.md` 才是真正生效的配置。
> 两者结构应该保持一致（章节、字段名、填写格式），否则模板会逐渐失去"新增 principal 时复制模板"的价值。
> **红线**：同步只动结构（章节/字段名/格式），绝不覆盖任何 principal 已填的真实值。

---

## 这个 SKILL 要解决的具体问题

在一次实际维护中发生过：编辑 `CONFIG_SAMPLE.md` 的"输出通道"字段时，只改了被问到的那一处，没有检查"这个改动是否该同步到已存在的 `CONFIG_TOWNEY.md` / `CONFIG_KLAIRE.md`"——导致改完之后模板和两个真实文件的格式不一致，且这个不一致没有被立刻发现。根因不是"故意区分对待"，而是改模板时缺一步"反向检查"。本 SKILL 就是把这一步流程化，避免再漏。

---

## 触发时机

- 即将编辑 `CONFIG_SAMPLE.md` 的任意章节（新增字段、删字段、改章节标题、改填写格式/示例）
- 发现某个 `CONFIG_{PRINCIPAL_ID}.md` 的章节结构跟 `CONFIG_SAMPLE.md` 不一致（无论是谁先发现的）
- 新增一个 principal，需要确认新文件是否完整覆盖了模板的必填章节

不适用于：只改某个 principal 自己的填值（如改 klaire 的 `annual_return`）——这种纯数值改动不涉及结构，不需要走本 SKILL。

---

## 操作步骤

### 第一步：列出所有真实配置档

真实配置档不再统一放在 `workspace-cio/`，而是分散在各 principal 前端 agent 的工作目录下：

```
Glob workspace-*/CONFIG_*.md，排除 workspace-cio/CONFIG_SAMPLE.md
```

不要硬编码"只有 towney 和 klaire"——principal 数量会变，每次都用 Glob 现查，避免漏掉后续新增的 principal。这张文件清单与 `custom-config-read` SKILL 里的 principal→路径对照表是同一份事实的两种呈现，新增 principal 时两处都要更新。

### 第二步：判断改动类型

对照 `CONFIG_SAMPLE.md` 改动前后的差异，分类：

| 改动类型 | 例子 | 处理方式 |
|---------|------|---------|
| 新增章节/字段 | 新增一个"xxx（必填）"章节 | 检查每个真实配置档是否需要补这一节；"必填"章节缺失要补，"可选"章节按该 principal 是否需要决定 |
| 删除章节/字段 | 本次删除"委员会构成"整节 | 同步删除每个真实配置档里对应的章节，不要只删模板 |
| 改填写格式（不改语义） | 输出通道从"两个分开字段"合并成"一个字段" | 把每个真实配置档现有的值**迁移**到新格式，不是清空重填——真实值不能丢 |
| 改章节的必填/可选标注 | "仓位体系"从必填改可选 | 不强制要求所有真实配置档都补这节；已有该节的可以保留，没有的不用补空节 |
| 模板新增/改了说明块文字 | "> 说明：..." 内容更新 | 同步复制到每个真实配置档对应章节下——说明块是给人看的引导文字，跟模板不一致会误导以后填新 principal 或回查字段含义的人 |

### 第三步：逐个真实配置档同步

对第一步列出的每个文件：

1. Read 该文件，定位对应章节
2. 按第二步的分类执行同步：删的删、补的补、格式迁移的迁移
3. **绝不**在同步过程中改动该章节里属于该 principal 自己的实际填值（如 `bitable_name`、`annual_return`、群聊 ID 等）——只动结构和格式，值本身原样保留或原样迁移到新格式里
4. 章节标题的"（必填）/（可选）"标注、以及该章节的"> 说明："引导文字，都跟模板保持一致——这两样是给人看的，不是要清理掉的模板残留，缺了反而会让以后维护的人看不出这节该填什么、是不是必填

### 第四步：完成后自检

逐个真实配置档过一遍以下检查清单：

- 模板里每个"必填"章节，该文件是否都有对应章节
- 该文件里是否还残留已被模板删除的旧章节/旧字段
- 该文件的字段名/格式是否跟模板当前版本一致（不要求章节顺序完全一致，但字段名必须一致，否则 `custom-config-read` 按字段名取值会取不到）
- 每个章节标题的"（必填）/（可选）"标注、和该章节的"> 说明："引导文字，是否跟模板当前版本一致（逐字对比，不要只凭印象觉得"应该差不多"）

### 第五步：如果某个真实配置档故意要跟模板不一样

允许真实配置档比模板少（不需要的可选章节直接没有），但**不允许**字段名不一致或多出模板里没有定义过的新章节——后者说明应该先反过来改模板，把这个新字段补进模板，再走一遍本流程同步到其他 principal，而不是只在一个 principal 的文件里悄悄加。

---

## 反向场景：先发现真实配置档跟模板不一致

不要默认"模板是对的，改真实文件"。先搞清楚谁该是权威：

- 如果这个不一致是因为模板更新了但忘记同步 → 走上面第三步同步真实文件
- 如果这个不一致是因为某个 principal 真的有合理的特殊需求，模板反而该被改 → 改模板，再走一遍同步到其他 principal
- 不确定是哪一种 → 问用户，不要自己猜原因（参考教训：曾经为了解释一个差异凑过"klaire 因为精简委员"这种没有依据的理由，被用户当场指出是编的）

---

## 设计原则（写新章节/字段时同样适用）

- 优先合并，而不是默认"业务规则不同就要分开字段"——能从值本身推断的区分信息（如 ID 前缀）不需要额外字段承载
- 不做没人用的字段。新增字段前确认确实有消费方（agent/SKILL 会读它），不要为"将来可能用到"预留配置项——没有用的字段只会增加用户填写成本，发现即删（参考已删除的 `active_members`，全仓库 grep 零消费方）
- 章节标必填还是可选，取决于"缺了这节会不会导致某个流程无法运行"，不是"大多数 principal 都填了"
